Dev is set up to run against a **local native PostgreSQL 17** (service `postgresql-x64-17`, `127.0.0.1:5432`, db `abode`, user/pass `postgres`/`postgres`), NOT the cloud Aurora cluster. `.env.local` `DATABASE_URL` points at localhost with `PGSSL=disable`; the Aurora URL is kept commented below it.

**Why:** Aurora `abode-dev` is IP-locked via its security group and unreachable when the user's public IP changes, and this machine has **no AWS CLI and no AWS credentials** (`AWS_PROFILE=abode` resolves to nothing, no `~/.aws/credentials`), so the infra scripts and any direct AWS SDK call (Cognito/S3) can't authenticate вЂ” the AWS SDK throws "Could not load credentials from any providers."

**How to apply:** For offline UI work, use the `/login` dev quick-login buttons (`ALLOW_DEV_LOGIN=true`) вЂ” they bypass Cognito and map an `abode_dev_user` cookie to a seeded row (owner = `owner@abode.dev`). **The `/signup` forms also work offline now:** the signup actions in `src/actions/auth.ts` honor the same `ALLOW_DEV_LOGIN` flag (`devAuthBypass()`/`createAccount`/`startSession`) вЂ” they skip Cognito, insert a local users row with a synthetic `dev:<uuid>` sub, and set the dev cookie. Both need a reachable, **seeded** DB: `npm run db:migrate && npm run db:seed` (these load `.env.local`). Restart `npm run dev` after any `.env.local` change. `loginAction` has a matching dev branch: with `ALLOW_DEV_LOGIN` on it logs in any existing local user **by email alone** (password ignored вЂ” no offline password store, same trust model as quick-login), so dev-created accounts can log back in. The shared `devBypass()` helper in `src/server/config.ts` (`NODE_ENV!=='production' && ALLOW_DEV_LOGIN==='true'`) gates these offline shortcuts; **tenant Dues "Pay" also works offline** via it вЂ” with no Stripe keys, `payInvoiceAction`в†’`payInvoiceDev()` records a payment and flips the invoice to paid, and the Dues **"Add a test card"** button (`addDevCardAction`в†’`addDevPaymentMethod()`, shown via the `devAdd` prop) inserts a fake Visa вЂў4242 вЂ” both without Stripe. Real Cognito signup/login, S3, and *real* card tokenization still need creds/IP restored; in production Cognito/Stripe are enforced for real (all dev branches are dead when `NODE_ENV=production`). The leaked root key still needs deleting per CLAUDE.md.
